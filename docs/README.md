### 2. Схема базы данных (DBML)
**Файл:** [dbdiagram.dbml](dbdiagram.dbml)

**Интерактивная визуализация:** [Открыть на dbdiagram.io](https://dbdiagram.io/d)

**Содержит:**
- ✅ 17 таблиц в формате DBML (Database Markup Language)
- ✅ Связи между таблицами (one-to-many, many-to-one)
- ✅ Primary Keys, Foreign Keys, Unique constraints
- ✅ Индексы на часто используемые поля
- ✅ Комментарии на русском для каждого поля
- ✅ ENUM типы для статусов и категорий
- ✅ Правильная нормализация (3NF)

**Структура таблиц:**
```
Управление пользователями: user (1 таблица)
Справочники: dictionary (1 таблица)
Контрагенты: contragent + contragent_contract + contragent_contact (3 таблицы)
Клиенты и Лиды (объединенные): client + client_contact (2 таблицы)
  • status = 'Lead': потенциальный клиент
  • status = 'Active': активный клиент
  • status = 'Inactive': неактивный клиент
Запросы: request + request_transport_unit + request_file + route_negotiation + route_negotiation_file (5 таблиц)
Маршруты: route_stage (универсальная таблица этапов маршрута, привязана к request) (1 таблица)
Финансы и документы: request_contract + request_document + request_invoice (универсальная: клиенты+контрагенты) + 
                     request_financial_record (4 таблицы)
```

**Примечание:** 
- Таблицы requests и orders объединены в одну таблицу request. Request со статусом New/Negotiating/Rejected = запрос, 
со статусом Approved/Completed = заказ. Для изменения маршрута Supervisor возвращает статус в Negotiating.
- Таблицы clients и leads объединены в одну таблицу client со статусом для разделения: Lead (потенциальные клиенты), 
Active/Inactive (клиенты). Контакты хранятся в единой таблице client_contact.
- Используется singular naming для таблиц (user, contragent, client, request) согласно современным conventions.

**Объем:** ~600 строк DBML кода  
**Визуализация:** Импортируйте файл на [dbdiagram.io](https://dbdiagram.io/d) для просмотра ER-диаграммы

## 🔄 Бизнес-процесс в двух словах

```
Лид/Клиент → Sales/Ops создает Запрос 
             ↓
             Логист согласует маршрут (чат)
             ↓
             Sales/Ops строит Окончательный Маршрут
             ↓
             Sales переводит Лида в Клиента (если новый клиент)
             ↓
             Sales/Ops формирует Заказ
             ↓
             Ops сопровождает Заказ:
             - Вводит фактические даты этапов
             - Загружает документы
             - Выставляет счета
             - Вводит доходы/расходы
             - При необходимости запрашивает изменения у Supervisor
             ↓
             Supervisor утверждает завершение
             ↓
             Заказ архивируется
```
